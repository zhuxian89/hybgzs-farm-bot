# 售卖功能 Bug 修复

## 🐛 问题描述

### 测试脚本问题
运行 `sell-now.js` 时，卖出了**全部库存**而不是保留 6 个。

原因：
1. 点击"全选"按钮后，自动填充了全部库存
2. 试图修改数量，但**作物匹配失败**（正则表达式不匹配页面格式）
3. 没有验证修改是否成功
4. 直接点击确认 → 卖出全部

### farm-bot.js 中的潜在问题
`setCropQuantityInQuickSell` 函数使用的正则表达式过于严格：

```javascript
// 旧的正则
cropName + '\\s+已选\\s+\\d+\\s*\\/\\s*库存\\s+(\\d+)'
// 期望格式：玉米 已选 0 / 库存 150
```

但实际页面格式可能：
- 有换行符
- 格式略有不同
- 导致匹配失败

## ✅ 修复内容

### 改进 `setCropQuantityInQuickSell`

**位置**: `scripts/farm-bot.js` 行 1587

**改进点**:
1. **添加备用匹配逻辑** - 如果严格匹配失败，使用宽松匹配
2. **添加 blur 事件** - 触发更多的表单事件确保生效

```javascript
// 改进的匹配逻辑：更宽松，支持不同的格式
// 匹配格式1: "玉米 已选 0 / 库存 150"
let match = text.match(new RegExp(cropName + '\\s+已选\\s+\\d+\\s*\\/\\s*库存\\s+(\\d+)'));

// 匹配格式2: 包含作物名和"库存"关键词（备用方案）
if (!match && text.includes(cropName) && text.includes('库存')) {
  const stockMatch = text.match(/库存\s*(\d+)/);
  if (stockMatch) {
    match = [null, stockMatch[1]];
  }
}
```

**关键改进**:
- 如果精确格式不匹配，只要文本包含作物名 + "库存" + 数字就认为找到了
- 添加 `blur` 事件触发，确保输入框失焦时触发验证

### 现有的保护机制（已存在）

farm-bot.js 中已经有多重验证，防止误卖：

1. ✅ **设置后验证** - `waitUntil` 等待数量设置生效
2. ✅ **二次确认** - 点击确认前再次读取选择状态
3. ✅ **结果验证** - 卖出后检查库存是否符合预期

这些机制即使匹配失败，也会抛出错误，**不会误卖**。

## 🔒 安全保障

修复后，即使正则匹配失败：
1. 设置数量会失败
2. `waitUntil` 验证会超时并抛出异常
3. 不会执行确认卖出
4. 整个售卖流程会失败，但**不会误卖全部库存**
