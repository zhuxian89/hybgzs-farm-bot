# 售卖功能核心 Bug 修复（完整版）

## 🐛 问题描述

用户反馈：**连续种植 4 轮南瓜，肯定收获成功了，但一次都没通知，导致一次都没进行卖出。**

### 问题 1：售卖失败
从日志看，售卖流程在执行，但验证失败：
```
[farm-bot] 南瓜持有 66，保留 6，准备卖出 60。
[farm-bot] 确认 确认卖出（1 种 · $70.92）
[farm-bot] 卖南瓜第 1/2 次失败：卖出后南瓜持有异常：期望 6，实际 66
```

售卖**看起来成功**，但验证时发现**库存没有变化**！

### 问题 2：收获无通知
即使收获成功，也**从来没有发送通知**，导致用户不知道已经收获，也不知道为什么没有卖出。

### 根本原因（两个问题同根）

**点击按钮后，立即执行验证/刷新操作，导致网络请求被中断！**

这是一个经典的 **Race Condition** - 脚本比网络快：

#### 问题 1：售卖请求被中断
代码流程：
```javascript
const label = await clickQuickSellConfirm(page);  // 1. 点击"确认卖出"
log(`确认 ${label}`);
await dismissNotice(page);                        // 2. 立即清除通知
await enterRecyclePage(page, `卖出后验证...`);   // 3. 立即导航到交易所
```

问题：
1. 点击按钮后，浏览器发送售卖请求到服务器
2. **但没有等待服务器响应**
3. 立即执行 `enterRecyclePage` → 导航到新页面
4. **导航会中断当前页面的所有请求**
5. 售卖请求被取消，服务器没有执行售卖
6. 验证时库存没变化 → 售卖失败

#### 问题 2：收获请求被中断
代码流程：
```javascript
button.click();                                    // 1. 点击"一键收获"
await dismissNotice(page);                         // 2. 立即清除通知
await waitUntil('收获操作完成，页面状态改变', ...) // 3. 立即读取状态
```

问题：
1. 点击按钮后，浏览器发送收获请求到服务器
2. **但没有等待服务器响应**
3. 立即执行 `dismissNotice` 和 `waitUntil` → 可能触发页面刷新
4. **收获请求可能被中断或返回旧状态**
5. 状态验证失败 → 收获被认为失败
6. 通知逻辑没有触发

## ✅ 修复方案

### 方案：在点击按钮后，等待 3 秒让请求完成

#### 修复 1：售卖功能

**修改**:
```javascript

const label = await clickQuickSellConfirm(page);
if (!label) {
  page.clearSpendConfirm();
  throw new Error('没有找到「确认卖出」按钮。');
}
log(`确认 ${label}`);
```

### 为什么是 3 秒？

1. **网络请求时间** - 发送请求 + 服务器处理 + 返回响应 ≈ 1-2 秒
2. **页面更新时间** - 浏览器更新 DOM ≈ 0.5 秒
3. **安全缓冲** - 网络波动/服务器慢 ≈ 0.5-1 秒
4. **总计** - 3 秒是安全且合理的等待时间

## ⚠️ 教训

**浏览器自动化的关键原则**：
1. **点击异步操作按钮后，必须等待操作完成**
2. **不要立即导航到其他页面或触发刷新**，会中断当前请求
3. **等待时间要充足**，考虑网络延迟和服务器处理时间
4. **收获和卖出都有同样的问题** - 这是系统性的设计缺陷

这是一个经典的 **race condition**：脚本速度快于网络请求，导致：
- ❌ 收获请求被中断 → 无通知
- ❌ 售卖请求被中断 → 库存不变

现在两个问题都已修复！
